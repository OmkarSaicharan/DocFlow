# Application Architecture Note

This document describes the architectural flow, database relationships, security model, and autosaving engine implemented in the **Collaborative Document Editor**.

---

## 2. Database Schema Design (Prisma)

The SQLite database schema resides in `/prisma/schema.prisma` and models a simple relational graph:

### A. `Document` Table
Stores primary rich-text documents:
- `id` (UUID): Primary key.
- `title` (String): Document header.
- `content` (String): Stored as plain HTML text generated by the TipTap editor.
- `ownerEmail` (String): Email of the document's creator. Used as the main ownership constraint.
- `createdAt` / `updatedAt` (DateTime): Auditing timestamps.

### B. `Share` Table
Stores collaborative access policies:
- `id` (UUID): Primary key.
- `documentId` (String): Foreign key referencing `Document.id`. Cascades on delete.
- `sharedWith` (String): Email of the user granted read/write access.
- **Unique Constraint**: `[documentId, sharedWith]` ensures no user is shared the same document multiple times.

---

## 3. Collaborative Access & Security Model

The backend Express route handlers (in `server.ts`) enforce rigorous permission validations:

1. **Access Guards**: For any action (Retrieving, Renaming, Editing, Saving, or Sharing), the request must provide the active user's email (`email` in query params or body).
2. **Owner Scope vs. Collaborator Scope**:
   - **Owners** can perform all CRUD operations, including viewing active shares, adding new share recipients, removing existing shares, and permanently deleting the document.
   - **Collaborators** can retrieve the document and edit its contents (updating text and headers), but are strictly blocked from editing sharing settings, viewing active share lists, or deleting the file.
3. **Self-Sharing Guard**: The backend strictly validates that users cannot share documents with themselves, returning a clear validation warning.

---

## 4. Debounced Autosave Engine

To prevent overwhelming the local disk and SQLite write queues on rapid keypresses, the client implements a dual-debounced engine:

* **Editor Text Updates**: Listens to TipTap's `onUpdate` event. Every keypress sets the status badge to `Saving changes...` and clears the active timer. A new 1000ms timeout is queued. When typing pauses for more than 1s, the content is securely synchronized via `PUT /api/documents/:id`.
* **Title Header Updates**: Listens to input changes in the title box. Employs a separate 800ms debounce timer to prevent database lockups while users are typing characters in-place.
* **Network Status Badge**: High-contrast, color-coded badges (`Saved`, `Saving changes...`, or `Autosave failed`) update dynamically. If a connection fails or database constraint triggers, the editor flags the failure with a descriptive inline diagnostic.
